# FEATURE: Group Mentorship Attendance & Admin Confirmation System

We have created the `group_mentorship_participants` table in the database with strict RLS (only `is_admin = true` can insert/delete, while admins, mentors, and `is_colaborador` can read). Now, let's implement the entire frontend and service ecosystem in parallel.

---

## 💾 1. SERVICE LAYER (src/lib/mentorshipService.ts)
Implement the following type and functions using the safe cast patterns:
1. **Export `GroupParticipant` interface:** `{ id: string; group_id: string; student_id: string; student_name: string; student_avatar?: string; created_at: string }`
2. **`addStudentToGroup(groupId: string, studentId: string): Promise<void>`**
   - Inserts a row into `group_mentorship_participants`.
   - Increments the `current_bookings` column by 1 on that `group_mentorships` row.
3. **`removeStudentFromGroup(groupId: string, studentId: string): Promise<void>`**
   - Deletes the corresponding row from `group_mentorship_participants`.
   - Decrements the `current_bookings` column by 1 on that `group_mentorships` row.
4. **`listGroupParticipants(groupId: string): Promise<GroupParticipant[]>`**
   - Queries `group_mentorship_participants` filtering by `group_id`.
   - Performs a join with the `profiles` table to retrieve the student's name and avatar.
